● Task
  • Receive a single raw English text (one or more sentences, punctuation may be missing).
  • Automatically detect sentence boundaries using punctuation, capitalization, and language heuristics.
  • Split the input into the smallest meaningful sentences you can. Think:
    – **One independent idea = one correction item.**
    – Whenever you see joins like commas, “and”, “but”, “because”, “so”, etc., **break them** into separate sentences—**unless** the part can't stand on its own.
    – If a sentence feels like a run-on or container for multiple thoughts, **cut it** into shorter pieces (aim for 1–2 clauses each).
    – **Hint** for splitting: Ask yourself, “Could this piece stand as a complete sentence if I added a period or question mark?** If yes, split it there.
    – But **don’t split** fixed or tight units—things like “New York City”, “take care of”, “5 kilometers”, titles, names, measurements, dates, or phrasal verbs.
  • All text should be split into sentences and no words or symbols should be left out.
  • Then analyze each resulting sentence for GRAMMAR, SPELLING, CAPITALIZATION, PUNCTUATION, and WORD-FORM errors.
  • There should be at most one punctuation mark at the end of each sentence. And wherever a punctuation mark is not needed semantically (such as and) or the sentence itself has a punctuation mark at the end, a punctuation mark should not be inserted.

● Output
  • Produce exactly one JSON object — no extra prose, no markdown, nothing outside the JSON.
  • The top-level JSON must use double quotes for keys and strings.
  • Schema:
    {
      "status": "OK",
      "corrections": [
        {
          "original": "<original sentence>",
          "changed": (true|false),
          "suggestion": "<tagged sentence>",
          "explanation": "short explanation"
        },
        …
      ]
    }
  • If you cannot parse the input at all, return:
    {
      "status": "ERROR",
      "message": "brief reason"
    }

● Correction tagging rules
  • For each sentence, compare original vs corrected.
  • If no change is needed: `"changed": false`, `"suggestion"` is the original sentence exactly, `"explanation": "no error"`.
  • If changes are needed: `"changed": true`, and in `"suggestion"` wrap only each exact wrong substring in `<wrong>…</wrong>` immediately followed by `<correct>…</correct>`.
    – Do not wrap the whole sentence or any extra context.
    – Combine grammar+spelling+capitalization errors into a single `<wrong>/<correct>` pair.
    – Punctuation fixes count as changes and should be tagged on the punctuation mark itself.
  • Spelling-only corrections: As an exception to the rule above, if a word has ONLY a spelling error, replace the word with its correct form inside a `<spell>…</spell>` tag. Grammatical errors always take precedence and must use the `<wrong>/<correct>` tags.
  • Do not nest tags or include any additional whitespace inside the tags.
● JSON validity
  • No trailing commas.
  • Ensure all tags (`<wrong>`/`</wrong>`, `<correct>`/`</correct>`, `<spell>`/`</spell>`) appear as literal text inside JSON strings.
  • Escape any internal double quotes using `\"`.
